敏捷开发模式

八月 12, 2019
  • 开发

一、敏捷开发的概念

1、需求经常改动
2、增量交付，迭代式开发、传统的瀑布式开发很难保证

二、敏捷开发人员的架构划分

1、po product owner 项目经理
2、sm scurm master 敏捷大师 也许就是科长级别的

3、team DEV 开发人员 QA 测试人员

三、敏捷开发的相关会议

任务明确
需求分析
故事点划分
故事点切割
迭代
每日立会：任务展板
+ 从昨天的立会到现在，我完成了什么？
+ 从现在到明天的立会，我计划完成什么？
+ 有什么阻碍了我的进展，风险和困难抛出来
敏捷回顾会：所有人都要参加进行回顾
+ 迭代速率、迭代燃起燃尽图
+ 迭代计划故事和实际完成的故事
+ 计划发布日期与实际发布日期
+ 客户满意度、团队满意度、生产环境的BUG数
+ 生产BUG解决时间、用户故事
平时写代码是什么样子的？任务是如何完成的？
+ 项目代码是拆分的
+ 立会上领取小的功能点
+ 两天到三天完成一个小功能
+ 完成自行验证 静态检查 复杂度 测试用例、功能测试
+ git push 提交代码 触发CI（持续集成）流程
+ 代码评审 code review
+ 人与人之间不影响 任务和任务之间不影响
+ 主分支的代码随时可以交付，不需要等大家都做完才可以交付，而且保证质量
